### 1.2 核心设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 后端语言 | **Node.js** | 异步 I/O 天然适合 SSE 长连接；VCPToolBox 同生态验证 |
| 框架 | **Express 4** | 生态成熟，中间件丰富；v5 正式发布后升级 |
| 插件模型 | **spawn 子进程为主，JS 插件可 require** | Python 插件必须子进程；Node.js 插件可同进程 |
| 工具协议 | **自定义文本协议 (YAML 块)** | 模型无关，任何 LLM 都能输出 YAML |
| 向量引擎 | **USearch via Rust N-API** | 千万级数据亚毫秒检索；编译为单个 .node 文件 |
| 关系存储 | **SQLite (better-sqlite3)** | 同步 API 简单可靠，单文件备份就是 cp |
| 前端策略 | **不绑定前端** | 兼容任何 OpenAI 格式的客户端 |

### 关键点说明

**为什么不是一次 LLM 调用就结束？**
AI 生成 `<<<TOOL>>>` 标记时，实际上是在说"我需要更多信息"。此时 AI 的回答只是一个"工具调用请求"，不是给用户的最终回复。所以正确的流程是：

```
用户消息 → LLM(第1次) → 检测到工具调用 → 执行工具
                                           ↓
用户 ← 返回最终回复 ← LLM(第2次) ← 注入工具结果
```

这个"多轮对话循环"在 `chat_handler.js` 中实现，默认最多循环 5 次，防止工具调用死循环。

## 3. 核心模块说明

### 3.1 server.js — 主入口

**职责**：应用初始化、中间件、路由挂载、生命周期管理

```
启动流程:
1. 加载 config.yaml (js-yaml)
2. 初始化 Express app
3. 注册中间件 (CORS, Auth, RequestID)
4. 初始化核心模块 (单例)
   - MemoryEngine.init()
   - PluginLoader.discover()
5. 挂载路由
6. 启动 WebSocket 服务
7. app.listen(port)

关闭流程:
1. 停止接收新请求 (DRAINING)
2. 等待进行中的请求完成 (最多 30s)
3. 持久化记忆索引
4. 关闭 WebSocket 连接
5. process.exit(0)
```

**代码量目标**：< 500 行

### 3.2 plugin_loader.js — 插件加载器

**职责**：扫描插件目录、解析 manifest、注册插件、生成工具列表

```javascript
// core/plugin_loader.js — 单例，启动时扫描 plugins/ 构建注册表
class PluginLoader {
    constructor() {
        this.plugins = new Map();  // name → { dir, manifest, config }
    }

    discover(globalPluginConfig) {
        // 扫描 plugins/ 下所有子目录，解析 manifest.yaml
        // 合并配置: manifest 默认值 → config.yaml 覆盖
    }

    getTools() {
        // 返回所有 type=tool 且 enabled 的插件，含 AI 可读参数描述
    }

    getPreprocessors() {
        // 返回所有 type=preprocessor 的插件
    }

    getStatics() {
        // 返回所有 type=static 的插件
    }

    reload(globalPluginConfig) {
        this.discover(globalPluginConfig);  // 重新扫描
    }
}

module.exports = new PluginLoader();  // 单例导出
```

**插件发现规则**：
- `plugins/<name>/manifest.yaml` 存在 → 识别为插件
- `manifest.yaml` 中 `enabled: false` → 跳过
- 目录名以 `_` 开头 → 跳过 (禁用标记)

**代码量目标**：< 300 行 | **实际**：~100 行

### 3.4 tool_protocol.js — 工具协议

**职责**：解析 AI 响应中的 YAML 块工具调用标记、生成给 AI 看的工具描述

**代码量目标**：< 250 行（含完整的单元测试）

详细规范见 [工具协议规范.md](./工具协议规范.md)

### 3.5 context_builder.js — 上下文装配

**职责**：在每次 LLM 调用前，组装完整的 system prompt 和 messages

```
装配顺序 (每一步的输出是下一步的输入):
1. 加载 base system prompt (来自角色配置)
2. 变量替换 ({{Date}}, {{Time}}, {{Today}} 等)
3. 记忆注入 (L3 标签/关键词召回结果，格式见记忆系统设计 §5)
4. 工具列表注入 (从 PluginLoader 获取)
5. 当前日期时间注入 (自动，防止 AI 瞎编日期)
6. 防幻觉准则注入 (自动)
7. 拼接历史消息 (最近 N 轮)
8. 追加当前用户消息
```

**关键实现细节**：
- 所有系统消息片段（systemPrompt + memories + toolPrompt + date + antiHallucination）在 `sysParts` 数组中拼接完成后，一次性 `join('\n\n')` 再 `messages.push()`。必须先拼接再 push，否则 JS 字符串不可变导致后续修改无法反映到已 push 的消息中。
- 当前日期格式：`YYYY年M月D日 星期X H:mm:ss（北京时间）`
- 防幻觉准则：不编造信息、不猜测用户信息、接受用户纠正

**代码量目标**：< 300 行

### 3.6 memory_engine.js — 记忆引擎

**职责**：记忆的写入、召回、修改、遗忘、标签管理

```javascript
class MemoryEngine {
    /**
     * 三层记忆管理（当前阶段 L3 已实现，L2 留给 Phase 4）：
     * L1: 短期上下文 (最近 N 轮对话)
     * L2: 中期记忆 (向量检索，Phase 4 Rust USearch)
     * L3: 长期记忆 (标签 + 关键词 LIKE 检索，SQLite)
     */
    async remember(entry) { /* 写入记忆 */ }
    async recall(query, topK = 5) { /* L3: 标签匹配 + 关键词 LIKE + 时间衰减 */ }
    async modify(id, updates) { /* 更新记忆，支持 content/summary/tags/importance */ }
    async forget(id) { /* 删除记忆 */ }
    formatForContext(memories) { /* 生成注入文本，附带 [mem_xxx] ID */ }
}
```

**当前实现状态**：
- L3 召回：中文双字/多字关键词 + CJK 单字兜底 + 标签 LIKE 匹配 + 时间衰减排序
- 混合评分：`标签匹配数 × 0.4 + importance × 0.4 + 召回频次 × 0.2`，再乘以时间衰减 `0.5^(天数/7)`
- 记忆注入：`- [mem_xxx] 内容` 格式，AI 可通过 ID 调用 `daily_note update` 修改

**向量引擎**：Phase 4 通过 Rust N-API (`rust-vector/`) 调用 USearch。

**代码量目标**：< 500 行

详细设计见 [记忆系统设计.md](./记忆系统设计.md)

### 3.9 modules/logger.js — 运行时日志

**职责**：统一的日志记录，支持分级、格式化、文件输出、内存缓冲

**API**：`{ createLogger, queryLogs, child }`

```javascript
// modules/logger.js — winston 封装
const { createLogger, queryLogs, child } = require('./modules/logger');

// 初始化（server.js 启动时调用一次）
const log = createLogger(config);

// 子 logger — 自动附加 module 字段
const modLog = child({ module: 'llm_client' });
modLog.info('LLM call -> openai:gpt-4o', { model: 'gpt-4o' });
// → "2026-06-08 23:00:00 INFO LLM call -> openai:gpt-4o | module:llm_client"

// 查询日志（管理面板用）
const results = queryLogs({ level: 'error', lines: 100 });
```

**实现**：
- `createLogger(config)` — 根据 `config.logging.level` + `config.logging.file` 创建 winston 实例（console + 可选文件 + 内存缓冲）
- `queryLogs({ level, lines, requestId })` — 查询内存中最近 5000 条日志
- `child(meta)` — 返回带预设字段的子 logger（`info/warn/error/debug` 方法自动附加 meta）
- 文件日志 10MB 轮转，最多 7 个文件

**日志级别使用规则**：

| 级别 | 用途 | 示例 |
|------|------|------|
| `error` | 需要立即关注的错误 | LLM API 调用失败、数据库损坏 |
| `warn` | 异常但可恢复 | 插件加载失败（跳过）、限流触发 |
| `info` | 关键业务流程 | 请求开始/结束、LLM 调用耗时、工具执行结果 |
| `debug` | 开发调试信息 | 完整请求体、SSE chunk 内容 |

**代码量目标**：< 80 行

## 4. 模块间依赖关系

```
server.js
  ├── modules/
  │   └── logger.js          (无依赖) ← 所有模块可用
  ├── core/
  │   ├── config.js          (无依赖) ← YAML 加载 + 安全检查
  │   ├── llm_client.js     (无依赖)
  │   ├── tool_protocol.js  (无依赖) ← 可独立测试
  │   ├── plugin_loader.js  (无依赖) ← 可独立测试
  │   ├── plugin_executor.js → plugin_loader, tool_protocol
  │   ├── memory_engine.js  → rust-vector (N-API)
  │   ├── context_builder.js → plugin_loader, memory_engine
  │   └── ws_server.js      (无依赖)
  ├── rust-vector/          → USearch C++ (Rust N-API, 独立编译)
  └── routes/
      └── chat_handler.js   → context_builder, llm_client, 
                               tool_protocol, plugin_executor
```

**设计原则**：
1. 核心模块尽量无依赖，每个文件可独立理解和测试
2. 路由层（chat_handler）是唯一的"编排者"，它组合各个核心模块
3. 模块间通过明确的接口（函数签名）通信，不直接访问彼此内部状态

## 4.1 上游 LLM 连接架构

Synapse 使用**单一地址模型**连接上游——只需 `api_base` + `api_key`，不区分"模式"。

```
┌──────────────────────────────────────────────────┐
│                上游 LLM 服务                       │
│                                                  │
│  OpenAI  │  NewAPI/one-api  │  Ollama  │  vLLM   │
│  官方 API │  聚合网关         │  本地模型  │  本地推理 │
└──────────────────────┬───────────────────────────┘
                       │  HTTP (OpenAI 兼容格式)
                       │  Anthropic 自动检测 (api_base 含 anthropic.com)
┌──────────────────────┴───────────────────────────┐
│                    Synapse                        │
│                                                  │
│  llm_client.js:                                  │
│  • isAnthropic(apiBase) → Messages API 格式       │
│  • 其他 → /chat/completions (OpenAI 兼容)         │
│  • 容灾: primary → fallback[0] → fallback[1] ...  │
└──────────────────────────────────────────────────┘
```

**为什么简化**：之前设计了三种模式（NewAPI/直连/本地），但实际使用中发现——无论 OpenAI 官方、NewAPI 聚合网关、Ollama 本地模型——它们都提供 OpenAI 兼容的 `/v1/chat/completions` 端点。唯一例外是 Anthropic（Messages API），通过检测 `api_base` 中是否包含 `anthropic.com` 自动处理。

**配置只需两行**：
```yaml
models:
  api_base: "https://api.openai.com/v1"  # 或 NewAPI/Ollama 地址
  api_key: "sk-your-key"
  primary: gpt-4o-mini                   # 默认模型名
```

## 5. 错误处理策略

```
异常分类:

1. 预期内异常 (可恢复)
   - 工具调用超时 → 返回错误给 AI，让 AI 决定重试或放弃
   - 插件返回错误 → 同上
   - LLM API 限流 → 指数退避重试 (最多 3 次)

2. 预期外异常 (不可恢复)
   - 数据库损坏 → 记录日志，返回 500
   - 内存溢出 → 优雅关闭

3. 用户侧异常
   - 无效请求 → 400
   - 鉴权失败 → 401
   - 超过速率限制 → 429

全局异常处理:
- Express 全局错误处理中间件统一捕获未处理异常
- 每个请求分配 request_id，便于日志追踪
- 生产环境不向客户端暴露内部错误详情
```

## 6. 性能考量

**个人使用场景的性能特征**：
- QPS < 1 (一个人聊天)
- 最慢的环节是上游 LLM API (秒级延迟)
- 本地的处理开销 (毫秒级) 完全可以忽略

**所以不需要优化的东西**：
- 数据库连接池 (单连接足够)
- 缓存层 (LLM 调用的响应本身就不可缓存)
- 消息队列 (没有流量尖刺)

**需要关注的东西**：
- SSE 连接不能阻塞 (用 async/await 保证)
- 大量历史消息时上下文装配的效率 (做了就行，不用过度优化)
- 向量索引文件大小 (定期清理过期记忆)

## 7. 安全考量

| 风险 | 防护措施 |
|------|----------|
| API Key 泄露 | config.yaml 不提交到 git (.gitignore) |
| 插件执行恶意代码 | 仅加载 plugins/ 下自己写的插件，不加第三方 |
| 工具调用注入攻击 | tool_protocol 解析器做严格的格式校验 |
| WebSocket 未授权连接 | 连接时验证 Token |
| 管理面板未授权访问 | HTTP Basic Auth |
| 日志中泄露敏感信息 | 自动脱敏 API Key 和用户内容 |

## 8. 代码规范

- **文件行数**：每个模块 < 500 行，路由处理函数 < 200 行
- **命名**：JavaScript 标准 camelCase，类名 PascalCase
- **类型**：使用 JSDoc 标注类型，核心模块可考虑 TypeScript
- **文档**：每个公开函数必须有 JSDoc 注释
- **日志**：使用 winston，关键节点打日志
- **配置**：所有可变参数走 config.yaml，不硬编码
- **异步**：统一使用 async/await，不用回调
