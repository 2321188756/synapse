## 一、架构合理性（确认无误，保持不变）

### 1.1 总体分层架构 ✅
```
客户端 → Synapse 中间层 → 上游 LLM
```
三层分离清晰，中间层作为透明代理——对上是标准 LLM API（OpenAI + Anthropic 兼容），对下是聚合网关/直连。

### 1.2 "插件即目录"设计 ✅
文件系统即注册中心。增删插件 = 增删文件夹，与 VCPToolBox 验证过的方案一致。

### 1.3 自定义文本工具协议 ✅
`<<<TOOL>>> ... <<<END>>>` 的 YAML 块协议：模型无关、人类可读、支持多行嵌套、StreamBuffer 处理流式截断。

### 1.4 三层记忆模型 ✅
L1（短期上下文）→ L2（中期向量）→ L3（长期标签+SQLite），含时间衰减、记忆巩固、标签共现。

### 1.5 三级模型容灾 ✅
```
primary → fallback[0] → fallback[1] → ... → 503
```

### 1.6 技术栈 ✅（已确认）
```
Node.js → 主服务、SSE 流式、WebSocket、路由
Rust    → 向量索引、Embedding 运算（N-API 绑定）
Python  → AI 插件子进程（spawn，stdio JSON 通信）
```

## 附录 A：技术选型对比

| 技术栈方案 | 优点 | 缺点 | 结论 |
|-----------|------|------|------|
| **Node.js + Rust + Python** | SSE 天然优势；VCPToolBox 同生态；性能最佳 | 三语言部署偏重 | ✅ **采用** |
| 纯 Node.js（hnswlib-node） | 单一运行时；部署简单 | 向量性能不如 Rust | 备选 |
| 纯 Python（FastAPI + ChromaDB） | 单语言；AI 生态全 | SSE 不如 Node.js | ❌ 放弃 |

## 附录 B：为何不选 ChromaDB

1. Node.js 主服务调用 ChromaDB 需额外起 Python 进程或 HTTP 服务，增加延迟和复杂度
2. 个人使用场景（~5000 条记忆），USearch 的 HNSW 索引比 ChromaDB 更轻量
3. Rust N-API 编译为单个 `.node` 文件，可分发给用户
4. VCPToolBox 已使用 Rust USearch 方案，经验可复用
