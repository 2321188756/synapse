## 一、发现的问题

### P1 — 建议修复（3 项）

#### 1.1 `core/llm_client.js` L21 — 冗余重复表达式

```javascript:21
const maxTokens = config.max_tokens || config.maxTokens || 4096;
```

`config.max_tokens` 写了两次，看起来是复制粘贴遗留，第二个本应是 `options.max_tokens` 之类的来源。当前行为正确（取 config 值，fallback 4096），但第二个 `config.maxTokens`（camelCase）若本意是 `options.maxTokens` 则会静默忽略。

**修复**：`config.max_tokens || 4096`（直接删掉重复项）

#### 1.2 `modules/logger.js` §3.9 与实现不一致

`架构设计.md` §3.9 的代码示例是简化版（直接 `module.exports = logger`），但实际 `modules/logger.js` 导出的是：

```javascript
module.exports = { createLogger, queryLogs, child, getLogger };
```

文档中未体现 `createLogger()` 工厂模式、`child()` 子 logger、`queryLogs()` 查询接口、`BufferTransport` 内存缓冲。

**修复**：更新 `架构设计.md` §3.9 的代码示例，反映实际 API。

#### 1.3 `server.js` L50 — `global.__synapse_config` 反模式

```javascript:50
global.__synapse_config = CONFIG;  // 供 health.js 等模块读取
```

`health.js` 通过 `global.__synapse_config` 获取配置，而 `models.js` 和 `chat_handler.js` 通过工厂函数参数传入 config。同一项目内两种不同的配置传递方式，不一致。

**建议**：将 health.js 也改为工厂函数模式：`createHealthRouter(config)`。或保持现状但加注释说明这是唯一的特例（health 路由不需要复杂的路由工厂）。

### P2 — 可延后（3 项）

#### 1.4 `routes/chat_handler.js` L71 — usage token 仍为 0

非流式响应 `usage: { prompt_tokens: 0, completion_tokens: 0, total_tokens: 0 }`。第一轮审计已标出，本轮仍未修复。

**建议**：Phase 2 工具协议实现时一并处理（上游 LLM 在最后 chunk 的 `usage` 字段返回真实值）。

#### 1.5 `config.example.yaml` 缺少 `system_prompt` 字段

`server.js` L54 支持 `CONFIG.system_prompt`，但配置模板中无此字段。第一轮已标出。

**建议**：加一行注释：
```yaml
# ---- System Prompt (可选，不设则用默认) ----
# system_prompt: "你是一个有用的 AI 助手。"
```

#### 1.6 `server.js` L11 — `config.env` 意图不明

```javascript:11
require('dotenv').config({ path: require('path').join(__dirname, 'config.env') });
```

`.gitignore` 中无 `config.env` 排除项，项目中也无此文件。如果为未来的环境变量覆盖预留，应加注释。如果不需要，应删除。

### 🟢 正面发现

#### 2.1 `health.js` 上游探测 — Phase 0 P3 提前实现

`/api/health` 现在返回上游 LLM 的连接状态：

```json
{
  "status": "ok",
  "version": "1.0.0",
  "uptime": 3600,
  "upstream": {
    "status": "ok",
    "provider": "api.openai.com",
    "latency_ms": 142
  }
}
```

这直接完成了 Phase 0 审计报告的 P3 项（"/api/health 返回上游 LLM 连接状态"）。

#### 2.2 `web/index.html` 健壮性显著提升

- `input` 事件即时保存 System Prompt（解决了 "点发送瞬间还没保存" 的 bug）
- `checkHealth()` 用 `/v1/models` + Bearer Token，与页面字段一致
- 连接检测失败自动重试 3 次（2s 间隔）
- apiBase/apiKey 变更 500ms 防抖自动重检

#### 2.3 config 简化 — 降低用户心智负担

从 "三种模式 + provider 字段" 简化为 "一个地址 + 一个 Key"。`primary` 和 `fallback` 从对象简化为字符串。这是一个优秀的产品决策。

#### 2.4 `start.bat` 英文 ASCII + 自动杀进程

解决了编码问题（cmd.exe GBK 乱码），新增端口占用自动 `taskkill`。
