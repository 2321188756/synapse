## 一、代码错误

### 🔴 1.1 `chat_handler.js` L98-L112 — 非流式路径重复 LLM 调用

```javascript:98-112
if (toolCallCount > 0) {
    // 非流式直接返回
    if (!stream) {
        const finalContent = await chat(messages, modelConfig, ...); // ← 重复调用！
        ...
    }
```

**问题**：工具循环最后一轮（L78）已经调用了 `chat()` 并得到了 `fullContent`（已包含工具结果合成的最终回复）。但非流式分支 L101 又调用了一次 `chat(messages, ...)`，此时 messages 没有包含最后一轮的 assistant 回复，LLM 会再次生成内容相同的回复。流式分支用 `fakeStream(fullContent)` 正确避免了重复，非流式分支也应该直接用 `fullContent`。

**修复**：
```javascript
if (!stream) {
    return res.json({ ... choices: [{ message: { content: fullContent } }] ... });
}
```

### 🟡 1.2 `memory_engine.js` L72 — 空关键词导致非法 SQL

```javascript:72
const likeClauses = keywords.map(() => '(tags LIKE ? OR content LIKE ?)').join(' OR ');
// 如果 keywords 为空 → likeClauses = '' → SQL 变成:
// SELECT * FROM memories WHERE ORDER BY ...  ← 语法错误
```

**触发条件**：用户发送纯标点或极短消息（如 "?"、"好"），`_extractKeywords` 返回空数组（中文 < 2 字不匹配，英文 < 3 字母不匹配），然后 L68 CJK 单字匹配也可能为空——但如果 query 有 CJK 字符则 L68 一定有值。

**实际风险评估**："好" → CJK 单字 `['好']` → 1 个 keyword。纯英文短输入 "ok" → 无 CJK、无英文 ≥3 字母 → 空数组 → L69 `query.length >= 2` → `keywords.push('ok')` → 1 个 keyword。真正的空关键词需要输入为空字符串或单个 ASCII 字符如 "?"。聊天场景罕见，但应防御。

**修复**：`if (keywords.length === 0) return [];`

### 🟡 1.3 `memory_engine.js` L66-L69 — CJK 单字 LIKE 全表扫描

```javascript:66-69
const cjk = query.match(/[一-鿿]/g) || [];
for (const ch of cjk) { if (!keywords.includes(ch)) keywords.push(ch); }
```

对于 20 个汉字的查询，产生 20 个 `%单字%` LIKE 条件。SQLite 的 `LIKE '%X%'` 无法使用索引，每条都是全表扫描 × 20 次。5000 条记忆时性能可接受（~10ms），但 SQL 字符串和参数数量膨胀。

**建议**：CJK 单字合并为一个 `OR` 子句扫描，或限制单字数量 ≤ 10。

### 🟢 1.4 `daily_note/main.py` L132-L133 — `cmd_search` 和 `cmd_list` 不可达

```python:122-126
def execute(params, config):
    cmd = params.get('command', 'create')
    if cmd == 'update': return cmd_update(params)
    return cmd_create(params)  # search/list 永远走不到
```

`cmd_search()`（L79-102）和 `cmd_list()`（L104-120）两个函数定义了但从未被 `execute()` 调用。manifest 中 `command` 参数的 `enum` 也只有 `[create, update]`。

**影响**：如果未来 manifest 扩展了 enum，这两个函数功能正常——但当前是死代码。

**建议**：如果 Phase 4+ 会用到，加 `# TODO: Phase 4 expose search/list commands`；如果不需要，删掉。

## 二、代码冗余

### 🟡 2.1 `chat_handler.js` L44 — 编码诊断日志残留

```javascript:43-44
const rawHex = Buffer.from(userMessage, 'utf8').toString('hex').slice(0, 80);
chatLog.info('--- REQ ' + requestId + ' --- user=' + userMessage.slice(0, 80) + ' | hex=' + rawHex, ...);
```

这是调试编码问题（GBK/UTF-8 混淆）时加的诊断代码。问题已修复，hex dump 不再需要。每次请求都做 hex 转换，且日志会打印完整用户消息（含敏感内容）。

**建议**：删除 hex dump 部分，或改为 `chatLog.debug(...)`。

### 🟡 2.2 `memory_engine.js` L64-L69 — 关键词提取做了两次去重工作

```javascript:64-69
const keywords = this._extractKeywords(query);    // 已有 new Set 去重
const cjk = query.match(/[一-鿿]/g) || [];
for (const ch of cjk) { if (!keywords.includes(ch)) keywords.push(ch); }  // O(n×m) includes
if (!keywords.includes(query) && query.length >= 2) keywords.push(query);
```

`keywords.includes()` 是 O(n) 扫描。两次循环中的 `includes` 可改用 Set。

**建议**：`const kwSet = new Set(keywords);` 然后 `kwSet.add(ch)` / `kwSet.add(query)`，最后 `[...kwSet]`。

### 🟢 2.3 `daily_note/main.py` L10 — DB_PATH 三重 dirname

```python:10
DB_PATH = os.path.join(os.path.dirname(os.path.dirname(os.path.dirname(__file__))), 'data', 'memory.db')
```

`__file__` = `plugins/daily_note/main.py`，三重 dirname 回到项目根。如果将来移动 daily_note 或改变插件目录结构，这个路径会断。

**建议**：在 manifest 的 config 中注入 `DATA_DIR`，或通过 stdin 传入。

### 🟢 2.4 `daily_note/main.py` — 直接操作 SQLite 与 memory_engine.js 逻辑重复

| 操作 | memory_engine.js | daily_note/main.py |
|------|:--:|:--:|
| INSERT memory | `remember()` L36-L40 | `cmd_create()` L34-L36 |
| INSERT tags | `remember()` L45-L48 | `cmd_create()` L37-L38 |
| UPDATE memory | `modify()` L160-L182 | `cmd_update()` L61-L73 |
| DELETE tags | `modify()` L169-L171 | `cmd_update()` L70-L73 |

daily_note 用 Python sqlite3 完全绕过了 memory_engine.js 的 Node.js API。意味着：
- `_updateCoOccurrence()` 永远不触发（标签共现表永远为空）
- `memory_engine.remember()` 在聊天流中从未被调用
- `memory_engine.modify()` 在聊天流中从未被调用
- 两套代码维护同一份 schema，容易不同步

**建议**：daily_note 不应直接写 SQLite。应该让 chat_handler 在工具执行后检测 `daily_note` 的返回结果，调用 `memoryEngine.remember()` / `memoryEngine.modify()` 写入。

### 🟢 2.5 `database.js` L16 — `fs` require 内联

```javascript:15-16
function init() {
    const fs = require('fs');  // ← 每次 init() 都 require，虽然会命中缓存
```

`fs` 是 Node.js 内置模块，应在文件顶部声明。

## 三、优化方向

### 🟡 3.1 `memory_engine.js` L72-L75 — SQL LIKE 构建缺少空值保护

`keywords` 可能为空（极短非中文输入），此时 SQL 的 WHERE 子句为空字符串 → 非法 SQL。参见 §1.2。

### 🟡 3.2 `memory_engine.js` L83-L88 — 召回算法与设计文档不一致

设计文档 `记忆系统设计.md` 4.2 节描述：
```
L2 向量检索权重 0.6, L3 标签检索权重 0.4
```
代码中 L2 未实现（Phase 4），仅有 L3 标签+关键词检索。打分公式：
```javascript
matched * 0.4 + importance * 0.4 + min(recallCount/10, 1) * 0.2
```
缺少设计文档中的时间衰减在教学公式中的位置（时间衰减在 L88 作为外部乘数 `score * timeDecay`）。当 60 天前的记忆得分再乘 `0.5^(60/7)` ≈ 0.0026 时，几乎永远不会被召回。

**建议**：短期降低衰减速度（如 30 天衰减 50%）或给时间衰减设下限。

### 🟢 3.3 `chat_handler.js` L176-L182 — fakeStream chunk 过小

```javascript:176-182
async function* fakeStream(text) {
    const chunkSize = 3;  // ← 每 3 字符一个 SSE chunk
    for (let i = 0; i < text.length; i += chunkSize) {
        yield { content: text.slice(i, Math.min(i + chunkSize, text.length)) };
    }
}
```

对于 1000 字符的回复，产生 334 个 SSE 消息。每个 SSE chunk 经过 `JSON.stringify` 后 ~200 字节，总计 ~67KB 网络传输（换行符 + `data:` 前缀）。虽然总量不大，但大量小消息增加了解析开销。

**建议**：`chunkSize = 10` 或 15。

### 🟢 3.4 `web/index.html` L403-L431 — 工具卡片 JSON 解析跨 chunk 缓冲

```javascript:404-431
if (delta.indexOf('🔧TOOL') !== -1 || toolBuf) {
    var src = toolBuf + delta;
    var markerStart = src.indexOf('🔧TOOL');
```

大括号计数法做 JSON 边界检测是正确的方向（比旧的 `split('|')` 强得多），但有几个边界问题：
- 如果 `🔧TOOL` 这个标记本身被 TCP 分包切断（如 `'🔧'` 和 `'TOOL{...}'` 在不同 chunk），`toolBuf` 会保存不完整的头部
- `braceCount` 逻辑假设 JSON 内没有嵌套对象中的 `{` `}`，但 preview 字段的内容是用户文本的转义版，不会出现（`\n` 被替换为 `\\n`）

**评估**：实际触发概率极低（SSE chunk 通常以完整行发送），不需要修改。

## 四、代码复用

### 🟡 4.1 `memory_engine.js` 与 `daily_note/main.py` — 双轨写入

这是 Phase 3 最大的架构问题（§2.4 已详述）。核心矛盾：

```
daily_note (Python) ──直接写──→ SQLite
memory_engine (Node) ──API 从未被调用──→ SQLite
```

两套代码维护同一 schema，标签共现表永不更新，时间衰减/recall 统计只在 Node 端 query 时才更新。

**建议**：在 `chat_handler.js` 中，`executeBatch` 返回结果后检测是否为 `daily_note` 的类型（或标准化为 "记忆写入" 操作），然后调用 `memoryEngine.remember()` / `memoryEngine.modify()`。

### 🟢 4.2 `routes/memories.js` — 只读 + 删除，无写入端点

当前 `/api/memories` 只暴露 `GET` 和 `DELETE`，没有 `POST`（创建）和 `PUT`（修改）。设计文档 `接口设计.md` §4.5-4.7 也只定义了 GET/DELETE/PUT，缺少 POST 是设计层面的选择（记忆由 AI 工具写入，不由管理 API 写入）。

**评估**：符合设计，不需要修改。

### 🟢 4.3 `core/config.js` — `get()` 使用率仍然很低

`config.get()` 仅被 `health.js`（上游探测）使用。`server.js` 用 `init()` 返回值、`chat_handler` 用工厂函数参数、`database.js` 不依赖 config。三种配置传递方式并存，但不影响功能。
