## 二、本轮新发现

本轮修改质量高，未引入新 bug。以下 2 项为优化建议，非问题：

### 🟢 2.1 `llm_client.js` L129 — `validateStatus: null` 的边界

选中 `null`（接受所有状态码）意味着 5xx 响应不会抛异常，而是走到 L132 手动判断。5xx 响应体可能是 HTML（nginx 错误页）而非 JSON/SSE 流，L135 `for await (chunk of resp.data)` 可能读取到 HTML 文本。功能上不会崩溃（HTML 文本也能 `toString()`），但错误信息会是一段 HTML。

**建议**：Phase 3 在 L134-138 加一句判断：如果 `content-type` 包含 `text/html`，直接抛 "Upstream returned HTML (likely nginx error page)"。

### 🟢 2.2 `chat_handler.js` L73 — `require` 可内联或模块级

```javascript:73
chatLog.info('  [round ' + round + '] tools parsed: ' + toolCalls.length, ...)
```

这里 `toolCalls` 来自 L72 的 `parseToolCalls(fullContent)`。每轮循环内调用，没问题。但在 L62 还有一个 `const { parseToolCalls, hasToolCalls, generateToolPrompt } = require('../core/tool_protocol');` 已经在模块顶部导入。无冗余。
