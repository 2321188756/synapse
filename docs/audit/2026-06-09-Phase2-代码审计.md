## 一、代码错误

### 🔴 1.1 `core/llm_client.js` L128-L130 — 重复 `validateStatus`，第一个被静默覆盖

```javascript:128-131
validateStatus: s => s < 500,   // ← 死代码，被下一行覆盖
family: 4,
validateStatus: null,            // ← 实际生效：接受所有状态码
```

JavaScript 对象字面量中同名字段后面的覆盖前面的。`s => s < 500` 永远不会执行。如果意图是接受所有状态码并自行处理，删掉第一个；如果意图是仅接受 < 500，删掉第二个。

**影响**：5xx 错误不会被 axios 抛出，而是走到 L133 的手动判断。功能上没问题（因 L133 有 `if (resp.status !== 200)` 兜底），但 5xx 的响应体可能不是合法 JSON/SSE 流，导致 L136 的 `for await` 或 L138 的 `safeStringify(resp.data)` 抛异常。**建议保留 `s => s < 500` 并删除第二个**。

### 🔴 1.2 `core/tool_protocol.js` L14 — 全局正则 `lastIndex` 跨调用污染

```javascript:14
const TOOL_RE = /<<<TOOL>>>\s*\n([\s\S]*?)<<<END>>>/g;
```

`RegExp.exec()` 带 `g` 标志会将匹配位置写入 `regex.lastIndex`。该属性在多次 `parseToolCalls()` 调用间持久存在。如果两次调用的文本长度满足特定条件（第一次长、第二次短但短文本长度 ≥ 前次 lastIndex），第二次调用会错过来自文本开头的匹配。

**复现条件**：第一次调用解析 500 字符的文本且匹配在末尾；第二次调用解析 200 字符的文本且工具块在前 200 字符内 → 第二次调用返回空。

**修复**：在 `parseToolCalls` 开头加 `TOOL_RE.lastIndex = 0;`。

### 🟡 1.3 `plugins/web_search/main.py` L72 — `or True` 使 else 分支死代码

```python:72-79
if "httpx" in sys.modules or True:  # 优先试 httpx
    try:
        import httpx
        data = search_with_httpx(query, max_results, api_key)
    except (ImportError, Exception):
        data = search_with_urllib(query, max_results, api_key)
else:
    data = search_with_urllib(query, max_results, api_key)  # ← 永远不会执行
```

`or True` 使条件恒为真。`else` 分支是死代码。看起来是调试残留——意图是 `if "httpx" in sys.modules:` 只在 httpx 已导入时才优先用它。

**修复**：删除 `or True`，恢复原始意图。或直接简化为 `try: httpx... except: urllib...`，因为 L12-15 已在模块级尝试了 httpx 导入。

### 🟢 1.4 `plugins/web_search/main.py` L33-34 — 函数内重复 import

```python:33-34

    import urllib.request   # ← 模块顶部 L14-15 已导入
```

`urllib.request` 和 `urllib.error` 已在文件顶部 L14-15 导入。函数内重复导入是无操作（Python 会命中 `sys.modules` 缓存），但多余且让人困惑。

## 二、代码冗余

### 🟡 2.1 `core/plugin_executor.js` L72-73 — 回调内内联 `require()`

```javascript:72-73
require('fs').appendFileSync(require('path').join(__dirname, '..', 'logs', 'plugin_error.log'),
```

`fs` 和 `path` 是 Node.js 内置模块，应在文件顶部声明。在 `child.on('close')` 回调内 `require` 每次都走模块缓存（开销可忽略），但会触发 ESLint `global-require` 警告，且降低可读性。`path` 已在 L10 导入，但这里的 `require('path')` 是多余的——直接用 `path.join(...)` 即可。

## 三、优化方向

### 🟡 3.1 `routes/chat_handler.js` L61-63 — 工具循环强制非流式

```javascript:61-63
for (let round = 0; round < MAX_TOOL_ROUNDS; round++) {
    fullContent = await chat(messages, modelConfig, { model, stream: false });
```

工具调用循环内部 L63 使用非流式 `chat()`（内部实际走 `stream: true` 收集，然后返回完整文本）。这意味着在工具执行期间，调试页面上用户看不到任何中间输出，直到所有工具调用完成 + 最终 LLM 流式回复。

**建议**：如果 LLM 输出了非工具内容（例如 "我来帮你搜索..."），可以在检测到 `<<<TOOL>>>` 之前先流式输出给用户。当前实现会缓存所有内容直到确认没有工具调用或循环结束。

### 🟢 3.2 `core/tool_protocol.js` L54 — 强制指令硬编码

```javascript:54
lines.unshift('', '【系统指令】当用户询问需要最新信息的问题时，你必须立即使用工具...');
```

这段中文指令与 `generateToolPrompt` 耦合。如果未来支持多语言或多角色配置，这条指令应该是可配置的。

### 🟢 3.3 `routes/chat_handler.js` L95 — token 估算粗糙

```javascript:95
usage: { prompt_tokens: Math.ceil(JSON.stringify(messages).length / 4), ... }
```

用 UTF-8 字节数 / 4 估算 token 数。对中文（3 字节/字，~0.5 token/字）会低估；对英文（1 字节/字，~0.25 token/字）会高估。误差可能在 2-3 倍。Phase 2 作为估算可以接受，但如果上游返回了真实 usage（OpenAI 在最后一个 SSE chunk 中），应优先使用。

## 四、代码复用

### 🟡 4.1 SSE 响应流模式重复

`chat_handler.js` 中有两个几乎相同的 SSE 流式响应处理流程：

1. **无工具调用直接流式**（L143-144 → `streamResponse`）
2. **工具有调用后最终流式**（L121-122 → `streamResponse`）

两者已通过 `streamResponse()` 函数共享。✅ 做得好。

但工具卡片的 SSE 注入（L115-119）与非工具调用的错误 SSE chunk（L149-156）使用了类似的 JSON 拼接模式。这些可以用 `sseWrite()` 统一——当前 `sseWrite` 已在 L164-170 定义但只在工具卡片场景使用。

### 🟡 4.2 `core/llm_client.js` L15-23 — `safeStringify` 未共享

`safeStringify` 是一个通用工具函数（安全 JSON 序列化，带 fallback），当前仅在 `llm_client.js` 内使用（2 处：L138、L206）。其他需要安全序列化日志的模块（如 `plugin_executor.js` L82）直接用了 `stdout.trim()`。

**建议**：如果后续其他模块也需要，提取到 `modules/` 下。Phase 2 范围内可以保留。

### 🟢 4.3 `core/config.js` L28 — `get()` 仅被 health.js 使用

```javascript:28-31
function get() {
    if (!_config) throw new Error('config not initialized — call init() first');
    return _config;
}
```

`get()` 只被 `health.js` L24 调用。`server.js` 直接使用 `init()` 的返回值，`chat_handler` 通过工厂函数参数接收 config。三种配置获取方式并存（直接返回值、工厂函数参数、`config.get()`），略有不一致。
