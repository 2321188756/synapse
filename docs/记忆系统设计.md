# 记忆系统设计

## 1. 设计目标

让 AI 在对话中"记住"过去的交互，并在未来对话中"回忆"起相关内容。

**三个核心原则**：
1. **渐进增强** — 即使没有记忆，系统也能正常工作。记忆是加分项，不是必需品
2. **成本可控** — 个人使用时，Embedding API 调用是主要成本来源，需要合理控制
3. **可解释** — 每次召回的结果要能看到来源，方便调试和信任

---

## 2. 三层记忆模型

```
┌─────────────────────────────────────────────────────────┐
│                    L1: 短期上下文                         │
│                                                          │
│  最近的 N 轮对话 (默认 20 轮)                              │
│  存储: 内存中的消息数组                                    │
│  召回: 直接拼接到 messages                                │
│  成本: 零                                                │
│  生命周期: 会话级 (关闭连接 = 清空)                        │
└─────────────────────────────────────────────────────────┘
                         │
                         │ 会话结束 / AI 标记重要
                         ▼
┌─────────────────────────────────────────────────────────┐
│                    L2: 中期记忆                           │
│                                                          │
│  最近 90 天的重要交互                                     │
│  存储: USearch 向量索引 (Rust N-API)                       │
│  召回: 语义相似度检索 (Embedding → Top-K)                  │
│  成本: 写入时一次 Embedding API 调用                       │
│  生命周期: 90 天 (过期自动清理)                            │
└─────────────────────────────────────────────────────────┘
                         │
                         │ 被多次召回 / 重要性高
                         ▼
┌─────────────────────────────────────────────────────────┐
│                    L3: 长期记忆                           │
│                                                          │
│  持久化的重要知识                                         │
│  存储: SQLite (标签索引 + 元数据)                          │
│  召回: 标签匹配 + 关键词检索                               │
│  成本: 零 (纯本地计算)                                    │
│  生命周期: 永久 (除非手动删除)                             │
└─────────────────────────────────────────────────────────┘
```

### 三层对比

| 维度 | L1 短期 | L2 中期 | L3 长期 |
|------|---------|---------|---------|
| **存储位置** | 内存 | USearch (Rust) | SQLite |
| **数据量级** | ~20 条消息 | ~5000 条记忆 | ~1000 条核心事实 |
| **检索方式** | 顺序拼接 | 向量相似度 | 标签 + 关键词 |
| **检索延迟** | 0ms | ~10ms | ~1ms |
| **写入成本** | 0 | 1 次 Embedding API | 0 |
| **丢失后果** | 重新开始对话 | 丢失近期上下文 | 丢失核心知识 |

## 3. 记忆数据结构

### 3.1 MemoryEntry — 统一记忆条目

```javascript
// 以下为数据结构定义，JavaScript 实现时使用 class + JSDoc
/**
 * @typedef {Object} MemoryEntry
 * @property {string} id              - 唯一标识 (UUID)

 * @property {string} createdAt       - ISO 时间戳

 * @property {number} recallCount     - 被召回的次数，默认 0
 * @property {string} [expiresAt]     - 过期时间 (L2 默认 90 天)
 * @property {number[]} [embedding]   - Embedding 向量
 */
```

### 3.2 SQLite 表结构 (L3)

```sql
-- 长期记忆主表
CREATE TABLE IF NOT EXISTS long_term_memories (
    id TEXT PRIMARY KEY,
    content TEXT NOT NULL,
    summary TEXT NOT NULL DEFAULT '',
    source TEXT NOT NULL DEFAULT 'conversation',
    conversation_id TEXT,
    tags TEXT NOT NULL DEFAULT '[]',        -- JSON 数组
    importance REAL NOT NULL DEFAULT 0.5,
    created_at TEXT NOT NULL,
    last_recalled_at TEXT,
    recall_count INTEGER NOT NULL DEFAULT 0,
    -- 不使用 expires_at，L3 是永久的
    UNIQUE(id)
);

-- 标签索引 (加速标签检索)
CREATE TABLE IF NOT EXISTS memory_tags (
    tag TEXT NOT NULL,
    memory_id TEXT NOT NULL,
    PRIMARY KEY (tag, memory_id),
    FOREIGN KEY (memory_id) REFERENCES long_term_memories(id) ON DELETE CASCADE
);

-- 标签共现表 (简化版 TagMemo，用于关联联想)
CREATE TABLE IF NOT EXISTS tag_co_occurrence (
    tag_a TEXT NOT NULL,
    tag_b TEXT NOT NULL,
    weight REAL NOT NULL DEFAULT 1.0,       -- 共现强度
    last_updated TEXT NOT NULL,
    PRIMARY KEY (tag_a, tag_b)
);

CREATE INDEX IF NOT EXISTS idx_memories_tags
    ON long_term_memories(tags);
CREATE INDEX IF NOT EXISTS idx_memories_importance
    ON long_term_memories(importance DESC);
CREATE INDEX IF NOT EXISTS idx_memories_created
    ON long_term_memories(created_at DESC);
CREATE INDEX IF NOT EXISTS idx_tag_co_occurrence_weight
    ON tag_co_occurrence(weight DESC);
```

## 4. 核心流程

### 4.1 记忆写入 (Remember)

```
触发条件:
  - AI 在回复中主动标记重要信息
  - 用户手动 "记住这个"
  - 每次对话结束后自动触发 (可选)

流程:
  用户: "我叫小王，最喜欢 Python 和 FastAPI"
    │
    ▼
  AI 判断 → 这条信息值得记住
    │
    ▼
  AI 输出:
    好的小王，我记住了你喜欢 Python 和 FastAPI。

    <<<TOOL>>>
    name: daily_note
    params:
      title: 用户信息 - 小王
      content: |
        用户名叫小王。
        最喜欢的编程语言: Python
        最喜欢的框架: FastAPI
      tags:
        - 用户信息
        - 偏好
        - 编程
      importance: 0.8
    <<<END>>>
    │
    ▼
  MemoryEngine.remember():
    1. 生成 content 的 Embedding
    2. 写入 USearch 向量索引 (L2 中期记忆，Rust N-API)
    3. 判断 importance ≥ 0.7 → 同时写入 SQLite (L3 长期记忆)
    4. 更新标签共现表
```

### 4.2 记忆召回 (Recall)

```
触发条件: 每次用户发送消息时自动触发

流程:
  用户: "帮我推荐一个 Web 框架"
    │
    ▼
  MemoryEngine.recall():
    │
    ├── L2 向量检索:
    │   1. 生成用户消息的 Embedding
    │   2. USearch 查询 Top-5 相似记忆 (Rust N-API)
    │   3. 记录相似度分数
    │
    ├── L3 标签检索:
    │   1. 从用户消息中提取关键词
    │   2. SQLite 查询匹配标签的记忆
    │   3. 按 importance × recency 排序
    │
    └── 合并结果:
        1. L2 结果权重 0.6, L3 结果权重 0.4
        2. 去重 (同一 content 去重)
        3. 按加权分数排序，返回 Top-5
        4. 更新 last_recalled_at 和 recall_count
    │
    ▼
  返回给 context_builder:
    ["用户叫小王，喜欢 Python 和 FastAPI", ...]
    │
    ▼
  注入 system prompt:
    """
    ## 相关记忆

    以下是与当前对话相关的历史信息，请参考：

    1. 用户叫小王，喜欢 Python 和 FastAPI。⭐ 重要度: 0.8
    2. 用户之前在学 Rust，后来觉得太难放弃了。重要度: 0.6

    ---
    """
```

### 4.3 记忆召回算法 (简化版)

```javascript
// core/memory_engine.js — 混合召回算法 (简化版)

class MemoryEngine {
    async recall(query, { topK = 5, l2Weight = 0.6, l3Weight = 0.4 } = {}) {
        const scored = new Map();  // id → { entry, score }

        // L2: USearch 向量检索 (通过 Rust N-API)
        const l2Results = await this._vectorSearch(query, topK * 2);
        for (const { entry, similarity } of l2Results) {
            // 时间衰减：7 天前的记忆衰减 50%
            const daysOld = this._daysSince(entry.createdAt);
            const timeDecay = 0.5 ** (daysOld / 7);
            scored.set(entry.id, {
                entry,
                score: similarity * l2Weight * timeDecay,
            });
        }

        // L3: SQLite 标签检索
        const keywords = this._extractKeywords(query);
        const l3Results = this._tagSearch(keywords, topK * 2);
        for (const entry of l3Results) {
            // 标签匹配数 × 重要度 × 召回频次
            const matchedTags = entry.tags.filter(t => keywords.includes(t)).length;
            const score = matchedTags * 0.3 + entry.importance * 0.5 +
                          Math.min(entry.recallCount / 10, 1.0) * 0.2;

            if (scored.has(entry.id)) {
                scored.get(entry.id).score += score * l3Weight;
            } else {
                scored.set(entry.id, { entry, score: score * l3Weight });
            }
        }

        // 排序 + Top-K
        const result = [...scored.values()]
            .sort((a, b) => b.score - a.score)
            .slice(0, topK)
            .map(item => item.entry);

        // 更新召回统计
        for (const entry of result) {
            await this._updateRecallStats(entry.id);
        }

        return result;
    }
}
```

## 5. 记忆注入格式（实际实现）

`formatForContext()` 生成格式（简化，不包含 importance/time，避免 token 浪费）：

```
## 已记录的信息

以下是关于用户的信息，这些是你已经知道的，直接引用即可，不需要再搜索：

- [mem_a1b2c3] 用户叫弦丝泪，中文名，英文名 XianSiLei
- [mem_d4e5f6] 用户喜欢 Python 和 FastAPI
```

**设计要点**：
- `[mem_xxx]` 附带记忆 ID，AI 可使用 `daily_note` 的 `command=update` + `id` 修改已有记忆
- 直接告诉 AI "这是已知信息，直接引用"，减少不必要的工具调用
- 不展示 importance/时间，节省 token

### 5.2 注入位置

在 system prompt 和工具列表之间注入：

```
[System Prompt 正文]

## 已记录的信息
...

【系统指令】...
## 可用工具
...
```

### 5.3 注入时机

每次用户消息到达时自动召回并注入。见 [架构设计.md §3.5](./架构设计.md)。

## 6. 记忆生命周期管理

### 6.1 自动清理

```javascript
// 定期清理任务 (每天凌晨 3 点由 node-schedule 触发)
async cleanExpiredMemories() {
    // L2: 删除超过 90 天的记忆 (从 USearch 索引)
    // L3: 检查 importance < 0.3 且 180 天未召回的长期记忆
    // 清理孤立的标签共现条目
}
```

### 6.2 记忆巩固 (Consolidation)

```javascript
// 记忆巩固 (每周执行一次)
async consolidate() {
    // 1. 查找 L2 中 recallCount ≥ 5 的记忆 → 提升到 L3
    // 2. 查找 L3 中标签高度重叠的记忆 (> 3 个共同标签) → 建议合并
}
```

**为什么需要巩固**：
- L2 有 90 天过期限制，反复被召回的记忆应该"永久化"
- 分散的碎片记忆 (每次对话一条) 需要合并成完整的知识

### 6.3 手动管理 (管理面板)

- **浏览**：按时间/标签/来源筛选记忆
- **编辑**：修改内容、标签、重要度
- **删除**：删除错误或过期的记忆
- **合并**：手动合并相关记忆
- **导出**：导出为 JSON/Markdown 备份

## 7. 隐私与安全

### 7.1 数据隔离

```
个人使用场景 → 只有一个用户 → 不需要多租户隔离
但设计上保留 user_id 字段，未来扩展用:
  CREATE TABLE ... user_id TEXT DEFAULT 'default'
```

### 7.2 敏感信息

```
AI 写入记忆前，检查是否包含:
  - API Key 格式 (sk-..., sk-ant-..., AIza...)
  - 身份证号格式
  - 银行卡号格式
  - 密码 / token / secret 关键词

检测到 → 自动拒绝写入，日志告警
```

## 8. 成本估算

### 个人使用的典型规模

| 指标 | 估算值 |
|------|--------|
| 每天对话轮次 | ~50 轮 |
| 每天新记忆数 | ~5 条 |
| Embedding 模型 | text-embedding-3-small (OpenAI) |
| 每条 Embedding 成本 | ~$0.00002 |
| **每天 Embedding 成本** | **$0.0001** |
| **每月 Embedding 成本** | **$0.003** |
| L2 存储量 (90 天) | ~5000 条记忆 → USearch 轻松承载 |
| L3 存储量 | ~200 条核心事实 → SQLite 一行代码的事 |

**结论**：个人使用场景，记忆系统的 API 成本可以忽略不计。
