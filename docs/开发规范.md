> **本文档是 [CLAUDE.md](../CLAUDE.md) 的详细展开。** Agent 首次进入项目先从 CLAUDE.md 开始，需要了解具体流程时再查阅本文档。

## 1. 文档同步详细流程

### 1.1 修改代码后的标准检查流程

```
修改了 file.js
    │
    ├→ 1. 查 CLAUDE.md 的"文档同步矩阵"
    │     找到对应的"必须检查"列表
    │
    ├→ 2. 逐一打开"必须检查"的文档
    │     搜索文档中引用该文件/模块/函数的地方
    │     例如: 搜索 "plugin_loader.js" 在架构设计.md 中出现的位置
    │
    ├→ 3. 判断是否需要更新
    │     ├→ 函数签名变了 → 必须更新代码示例
    │     ├→ 新增了参数 → 必须在参数说明中补充
    │     ├→ 行为逻辑变了 → 必须更新文字描述
    │     └→ 仅内部重构，外部接口不变 → 不需要更新文档
    │
    └→ 4. 需要更新 → 执行更新，在变更说明中引用本规范
```

### 1.2 文档间交叉依赖

```
README.md
  ├── 引用 docs/架构设计.md（项目结构部分 ∮1）
  ├── 引用 docs/插件系统规范.md（内置插件表 ∮8）
  └── 路线图 checkbox ← Phase 完成时必须更新

docs/架构设计.md
  ├── 引用 docs/工具协议规范.md（∮3.4）
  ├── 引用 docs/记忆系统设计.md（∮3.6）
  └── 路由定义 ← 与 docs/接口设计.md 保持一致

docs/接口设计.md
  ├── ∮2.3 引用 core/protocol_bridge.js（文件名必须正确）
  └── ∮5 模型路由逻辑 ← 与 core/llm_client.js 保持一致

docs/记忆系统设计.md
  ├── 数据结构和召回算法 ← 与 core/memory_engine.js 保持一致
  └── rust-vector/ ← 与 Cargo.toml 保持一致

docs/插件系统规范.md
  └── manifest.yaml 规范 ← 是所有 plugins/*/manifest.yaml 的权威定义

docs/audit/*.md
  └── 追溯清单 ← 指向所有其他文档的具体行号，需定期验证

config.example.yaml
  └── 注释 ← 与 docs/部署指南.md 的配置说明保持一致
```

## 2. 开发日志撰写标准

### 2.1 日志文件命名

```
devlog/001-phase0-项目初始化.md     ← Phase 级日志
devlog/002-phase1-后端骨架.md
devlog/003-phase2-工具协议.md
```

如果一个 Phase 特别大需要拆分，用子编号：
```
devlog/004a-phase3a-向量写入.md
devlog/004b-phase3b-向量检索.md
```

### 2.2 日志模板

每篇日志严格遵循 `devlog/000-模板.md` 的结构：

```markdown
# 开发日志 #XXX — [标题]

**日期**：YYYY-MM-DD
**状态**：[进行中 / 已完成]

---

## 完成内容

### [功能模块名]
- 具体改动 1
- 具体改动 2

### 文件变更
| 文件 | 操作 | 说明 |
|------|------|------|
| `path/to/file` | 新增/修改/删除 | 说明 |

---

## 遇到的问题

### 问题 1：[描述]
- **原因**：
- **解决**：

---

## 关键决策

- 决策 1：[说明 + 理由]

---

## 下一步
- [ ] 待办 1
```

### 2.3 日志粒度规则

| 变更类型 | 记录方式 |
|---------|---------|
| 新建文件（>50 行） | 必须记录在"文件变更"表 |
| 修改文件（>20 行或行为变更） | 必须记录 |
| 删除文件 | 必须记录 |
| 修正错别字 / 格式化 | 不记录 |
| 设计决策（改变方向） | 必须记录在"关键决策"章节 |
| 修复 bug | 记录在"遇到的问题"章节 |

## 3. 架构审计流程

### 3.1 何时执行审计

| 触发时机 | 审计深度 | 输出 |
|---------|:--:|------|
| **每个 Phase 完成后** | 全面审计 | 更新 `docs/audit/` 下的现有报告，或新建报告 |
| **重大设计变更** | 专项审计 | 新建 `docs/audit/YYYY-MM-DD-变更描述-审计.md` |
| **发现文档-代码不一致** | 快速追溯 | 修正不一致的文档，在审计报告中标注 |

### 3.2 审计检查清单

执行架构审计时，按以下清单逐项检查：

- [ ] README.md 路线图 checkbox 与 devlog 状态一致
- [ ] CLAUDE.md 文档同步矩阵覆盖所有 docs/ 文件
- [ ] docs/架构设计.md 模块列表与 core/ 目录文件一致
- [ ] docs/接口设计.md 端点与 routes/ 文件一致
- [ ] docs/记忆系统设计.md 数据结构与 memory_engine.js 一致
- [ ] docs/工具协议规范.md 协议格式与 tool_protocol.js 一致
- [ ] docs/插件系统规范.md manifest 定义与 plugin_loader.js 一致
- [ ] 所有代码示例的语言标注与实际技术栈一致（JS ≠ Python）
- [ ] 所有文件名引用（如 `core/xxx.js`）在文档中是正确的
- [ ] config.example.yaml 所有新增配置项在部署指南.md 有说明
- [ ] 没有多个文档重复维护同一份信息

### 3.3 审计报告格式

```markdown
# Synapse 架构审计报告

**审计人**：项目架构师
**审计日期**：YYYY-MM-DD
**项目阶段**：Phase X
**审计范围**：[全量 / 专项：xxx]

---

## 整改总结

| 优先级 | 发现 | 已修复 | 剩余 |
|--------|:---:|:----:|:---:|
| 🔴 P0 | N | N | N |

---

## 一、发现问题

### P0 — 严重（阻塞开发）
[文件:行号] 问题描述 → 修复方案

### P1 — 中等（建议修复）
### P2 — 轻微（可延后）
### P3 — 增强（Phase 2+）

---

## 二、往期问题追溯

列出上一轮审计的所有问题及其当前状态（已验证 / 未修复 / 部分修复）。

---

## 三、整改清单

- [ ] R1 ...
```

**关键规则**：审计报告的"往期问题追溯"章节**必须逐条验证**上一轮报告中的每个问题——不能假设未修复，必须实际读代码确认。

## 4. 命名规范

### 4.1 文件

| 类型 | 规范 | 示例 |
|------|------|------|
| JavaScript 模块 | snake_case | `plugin_loader.js`、`context_builder.js` |
| 设计文档（中文） | 中文名 | `架构设计.md`、`接口设计.md` |
| 插件目录 | snake_case | `web_search/`、`daily_note/` |
| 插件入口 | `main.py` 或 `main.js` | `plugins/web_search/main.py` |
| 开发日志 | `XXX-phaseN-描述.md` | `001-phase0-项目初始化.md` |
| 审计报告 | `YYYY-MM-DD-描述-审计.md` | `2026-06-08-架构审计报告.md` |

### 4.2 代码

- JavaScript：camelCase 变量/函数，PascalCase 类名
- Python：snake_case 变量/函数，PascalCase 类名
- Rust：snake_case 变量/函数，PascalCase 类型

### 4.3 YAML 字段

- 统一 snake_case：`display_name`、`max_tokens`

## 5. 代码风格

### 5.1 文件行数

| 文件类型 | 上限 | 说明 |
|---------|:--:|------|
| 核心模块（core/*.js） | 500 行 | 超出则拆分为多个模块 |
| 路由处理函数 | 200 行 | 逻辑下沉到 core 模块 |
| 插件 main.py | 200 行 | 复杂逻辑抽到插件的 lib/ 子目录 |
| 设计文档 | 无硬限制 | 以"一个主题一篇"为原则 |

### 5.2 注释

- 每个公开函数必须有 JSDoc 注释
- 使用 `//` 注释，不用 `#`（Python 文件除外）
- 文档中的代码示例必须标注语言：````javascript` / ````python` / ````yaml`

### 5.3 其他

- 统一使用 `async/await`，不用回调
- 所有可变参数走 `config.yaml`，不硬编码
- 日志使用 `winston`
- 配置项命名用 snake_case：`api_key`、`l2_expire_days`

## 6. 插件开发检查清单

开发新插件前，逐项确认：

- [ ] 确定插件类型：tool / static / preprocessor
- [ ] 创建目录 `plugins/<snake_case_name>/`
- [ ] 写 `manifest.yaml`（参照 `docs/插件系统规范.md §3`）
- [ ] 写入口代码（参照 `docs/插件系统规范.md §4`）
- [ ] 参数校验（required 字段缺失时返回友好错误）
- [ ] 超时处理（默认 30 秒）
- [ ] 正常调用测试
- [ ] 参数缺失测试
- [ ] 上游 API 不可用测试
- [ ] 在 `docs/插件系统规范.md §8` 注册
- [ ] 在 README.md 项目结构中更新 `plugins/` 子目录

## 7. Phase 收尾检查清单

一个 Phase 完成、准备标记"✅ 完成"前：

- [ ] devlog 已写（日期、完成内容、文件变更表、关键决策、下一步）
- [ ] devlog 状态已改为"✅ 完成"
- [ ] README.md 路线图对应 checkbox 已勾选
- [ ] docs/audit/ 审计报告已更新（或确认无需更新）
- [ ] CLAUDE.md 的同步矩阵仍然准确（新增了模块吗？）
- [ ] 所有文档中的代码示例语言标注正确
- [ ] 所有文件名引用正确（`.py` vs `.js`）
