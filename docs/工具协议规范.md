# 工具协议规范

> **注意**：本文档中的代码示例为语言无关的伪代码，旨在说明算法逻辑。实际实现使用 Node.js/JavaScript（`core/tool_protocol.js`）。

## 1. 设计目标

让 AI 用**自然文本**表达工具调用意图，而不依赖任何特定 LLM 的 Function Calling 实现。

**核心优势**：
- **模型无关**：从 OpenAI 换到 Claude 换到 Gemini 换到本地模型，工具调用语法不变
- **人类可读**：不依赖 JSON Schema，AI 的输出可以直接阅读和调试
- **可嵌套**：支持在工具调用参数中嵌入复杂内容（代码、文档等）
- **向前兼容**：新增高级特性（异步、流式）不破坏已有插件

## 2. 协议格式

### 2.1 基本语法

```
<<<TOOL>>>
name: <插件名称>
params:
  <参数1>: <值1>
  <参数2>: <值2>
  <参数3>: |
    <多行值>
<<<END>>>
```

**规则**：
- 以 `<<<TOOL>>>` 开头，`<<<END>>>` 结尾
- 内部使用 **YAML 格式**
- `name` 字段指定插件名称
- `params` 字段包含传递给插件的参数

### 2.2 完整示例

```
<<<TOOL>>>
name: web_search
params:
  query: FastAPI SSE streaming tutorial 2026
  engine: tavily
  max_results: 5
<<<END>>>
```

### 2.3 多行参数

用 YAML 的 `|` (保留换行) 或 `>` (折叠换行) 语法：

```
<<<TOOL>>>
name: daily_note
params:
  title: 架构设计讨论记录
  content: |
    今天讨论了 Synapse 的架构设计：

    1. 前端使用任意 OpenAI 兼容客户端
    2. 后端使用 Node.js + Express
    3. 记忆系统用 USearch (Rust) + SQLite

    决定先用 Phase 1 的 MVP，后续迭代加功能。
  tags:
    - 架构
    - AI
    - 开发日志
<<<END>>>
```

### 2.4 多个工具调用

在一次响应中可以包含多个工具调用块：

```
我来帮你搜索一下，然后保存到日记里。

<<<TOOL>>>
name: web_search
params:
  query: FastAPI best practices
  engine: tavily
<<<END>>>

<<<TOOL>>>
name: daily_note
params:
  title: FastAPI 最佳实践
  content: |
    # 搜索结果整理
    等待搜索结果...
  tags:
    - FastAPI
    - 学习笔记
<<<END>>>
```

**解析规则**：按出现顺序依次执行。如果有依赖关系（第二个工具需要第一个工具的结果），AI 应该分两次对话发送，而不是放在同一个响应中。

## 3. 解析算法

```python
import re
import yaml
from dataclasses import dataclass, field
from typing import Any

TOOL_PATTERN = re.compile(
    r'<<<TOOL>>>\s*\n(.*?)\n<<<END>>>',
    re.DOTALL
)

@dataclass
class ToolCall:
    name: str
    params: dict[str, Any] = field(default_factory=dict)
    raw_text: str = ""              # 原始文本 (调试用)
    position: int = 0               # 在响应中的位置

def parse_tool_calls(text: str) -> list[ToolCall]:
    """
    从 AI 响应文本中提取所有工具调用。

    Args:
        text: AI 的完整响应文本

    Returns:
        解析出的工具调用列表，按出现顺序排列。
        如果响应中没有工具调用，返回空列表。
    """
    matches = TOOL_PATTERN.finditer(text)
    calls = []

    for idx, match in enumerate(matches):
        yaml_text = match.group(1).strip()
        try:
            data = yaml.safe_load(yaml_text)
            if not isinstance(data, dict) or "name" not in data:
                continue
            calls.append(ToolCall(
                name=data["name"],
                params=data.get("params", {}),
                raw_text=match.group(0),
                position=idx,
            ))
        except yaml.YAMLError:
            # 解析失败 → 跳过这个块，不影响其他块
            continue

    return calls

def has_tool_calls(text: str) -> bool:
    """快速检查响应中是否包含工具调用"""
    return "<<"<TOOL>>>" in text
```

### 3.1 容错设计

AI 输出的 YAML 可能不完美，解析器需要容忍：

| 问题 | 处理方式 |
|------|----------|
| YAML 语法错误 | 跳过该块，不影响其他块 |
| 缺少 `name` 字段 | 跳过该块 |
| `params` 不存在 | 默认为空字典 |
| 缩进不一致 | YAML 库自带容错，一般情况能处理 |
| `<<<TOOL>>>` 不完整 (流式截断) | 在 SSE 流式场景中做缓冲拼接 |

### 3.2 流式场景处理

LLM 是流式输出的，`<<<TOOL>>>` 块可能跨多个 SSE chunk：

```python
class StreamBuffer:
    """
    流式输出的缓冲器，解决工具调用块跨 chunk 的问题。
    """
    def __init__(self):
        self.buffer = ""
        self.in_tool_block = False

    def feed(self, chunk: str) -> list[str]:
        """
        喂入一个 chunk，返回可以安全发给客户端的文本。
        如果正在工具调用块内部，返回空字符串 (暂扣)。
        """
        self.buffer += chunk

        # 检测工具块开始
        if "<<<TOOL>>>" in self.buffer and not self.in_tool_block:
            self.in_tool_block = True
            # 返回 <<<TOOL>>> 之前的文本
            before, self.buffer = self.buffer.split("<<<TOOL>>>", 1)
            self.buffer = "<<<TOOL>>>" + self.buffer
            return [before] if before.strip() else []

        # 检测工具块结束
        if self.in_tool_block and "<<<END>>>" in self.buffer:
            self.in_tool_block = False
            # 此时 buffer 包含完整的工具调用块
            # 继续缓存，等完整响应结束后统一解析
            return []

        # 不在工具块内 → 直接透传
        if not self.in_tool_block:
            out = self.buffer
            self.buffer = ""
            return [out] if out.strip() else []

        return []

    def flush(self) -> str:
        """获取缓冲区中剩余内容"""
        return self.buffer
```

## 4. 生成给 AI 看的工具描述

`plugin_loader.js` 负责将所有已注册工具的 manifest 信息转换为 AI 可读的文本，注入 system prompt。

### 4.1 生成格式

```python
def generate_tool_prompt(plugins: list[Plugin]) -> str:
    """
    根据已注册的工具插件，生成注入 system prompt 的工具描述。
    """
    lines = ["## 可用工具\n"]
    lines.append("你可以使用以下工具来完成任务。")
    lines.append("调用工具时，使用以下格式：\n")
    lines.append("```")
    lines.append("<<<TOOL>>>")
    lines.append("name: <工具名称>")
    lines.append("params:")
    lines.append("  <参数>: <值>")
    lines.append("<<<END>>>")
    lines.append("```\n")

    for plugin in plugins:
        if plugin.type != "tool":
            continue

        tool = plugin.tool_config
        lines.append(f"### {plugin.display_name} (`{plugin.name}`)\n")
        lines.append(tool["instruction"].strip())
        lines.append("")

        # 参数列表
        if tool.get("parameters"):
            lines.append("**参数：**")
            for param_name, param_def in tool["parameters"].items():
                required = "必填" if param_def.get("required") else "可选"
                default = f"，默认值: {param_def['default']}" if "default" in param_def else ""
                desc = param_def.get("description", "")
                lines.append(f"- `{param_name}` ({param_def['type']}, {required}{default}): {desc}")
            lines.append("")

        # 调用示例
        if tool.get("examples"):
            lines.append("**示例：**")
            for ex in tool["examples"]:
                lines.append(f"// {ex['description']}")
                lines.append("```")
                lines.append(ex["call"].strip())
                lines.append("```")
                lines.append("")

    return "\n".join(lines)
```

## 5. 与 OpenAI Function Calling 的对比

| 维度 | 自定义文本协议 | OpenAI Function Calling |
|------|:---:|:---:|
| **模型兼容性** | ✅ 任何能输出文本的模型 | ❌ 仅 OpenAI/兼容模型 |
| **实现复杂度** | 200 行解析器 + YAML 库 | 标准 JSON Schema |
| **AI 学习成本** | ⚠️ 需要在 system prompt 中教 | ✅ 标准格式，模型已预训练 |
| **嵌套能力** | ✅ YAML 多行文本天然支持 | ⚠️ JSON 转义繁琐 |
| **流式解析** | ⚠️ 需要 StreamBuffer 缓冲 | ✅ JSON 天然分块 |
| **错误调试** | ✅ 人类可读，直接看 AI 输出 | ⚠️ JSON 可读性差 |
| **生态工具** | ❌ 需要自己写一切 | ✅ 丰富的 JSON Schema 工具链 |

### 建议的双模策略

如果你主要用 OpenAI/Claude (这两个都支持 Function Calling)，可以**双模运行**：

```python
# core/tool_protocol.js

def parse_tool_calls(text: str, openai_tool_calls: list = None) -> list[ToolCall]:
    """
    双模解析：
    1. 优先用标准 Function Calling 结果 (如果可用)
    2. 回退到自定义文本协议解析
    """
    if openai_tool_calls:
        return [ToolCall(name=tc.function.name,
                        params=json.loads(tc.function.arguments))
                for tc in openai_tool_calls]
    return parse_from_text(text)
```

这样你用 OpenAI 时享受标准 Function Calling 的可靠性，切换到不支持 Function Calling 的模型时自动回退到文本协议。

## 6. 错误处理

### 6.1 AI 调用了不存在的工具

```
AI: <<<TOOL>>> name: non_existent_tool ... <<<END>>>
```

**处理**：返回友好错误，让 AI 修正

```python
error_result = ToolResult(
    status="error",
    error=f"工具 'non_existent_tool' 不存在。可用工具: {', '.join(available_names)}",
    message_for_ai=f"工具名称错误。可用工具: {', '.join(available_names)}"
)
```

### 6.2 参数缺失或类型错误

```
AI: <<<TOOL>>> name: web_search params: query: "" <<<END>>>
```

**处理**：根据 manifest 中的定义做参数校验

```python
def validate_params(tool_call: ToolCall, param_defs: dict) -> list[str]:
    """校验参数，返回错误列表。空列表 = 通过"""
    errors = []
    for name, definition in param_defs.items():
        if definition.get("required") and name not in tool_call.params:
            errors.append(f"缺少必填参数: {name}")
        # 类型检查、范围检查等...
    return errors
```

### 6.3 工具执行超时

```python
async def execute_with_timeout(plugin, params, timeout: int = 30):
    try:
        return await asyncio.wait_for(
            plugin.execute(params),
            timeout=timeout
        )
    except asyncio.TimeoutError:
        return ToolResult(
            status="error",
            error=f"工具执行超时 ({timeout}s)",
            message_for_ai=f"工具 {plugin.name} 执行超时，请尝试缩小查询范围或稍后重试。"
        )
```
